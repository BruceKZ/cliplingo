# 桌面翻译客户端产品说明书与开发指南

## 1. 文档目的

本文档定义一款轻量级桌面文本翻译客户端的产品需求、技术栈、系统架构、界面规范、安全约束、开发规范与交付标准。

本项目目标不是构建通用翻译平台，也不是集成大量非核心能力，而是实现一个可随时调用的、极简、轻量、跨平台的文本翻译工具。

本文档中的“必须”“禁止”“仅允许”均为强制要求。

---

## 2. 产品定位

### 2.1 产品目标

构建一款适用于 macOS 与 Windows 的桌面端翻译客户端，满足以下核心链路：

**用户选中文本 → 连续两次复制（Cmd/Ctrl+C+C）→ 自动读取剪贴板 → 调用可配置 AI API 翻译 → 快速弹出结果窗口显示译文**

### 2.2 产品边界

本产品仅处理文本翻译，不承担以下职责：

* 不实现 OCR
* 不实现截图翻译
* 不实现网页全文翻译
* 不实现浏览器插件
* 不实现文档批量翻译
* 不实现 TTS 朗读
* 不实现账号系统
* 不实现云同步
* 不实现术语库云管理
* 不实现团队协作

允许保留少量本地历史记录与本地配置。

### 2.3 设计原则

* 极简
* 快速
* 低资源占用
* 可自定义 API
* 安全优先
* 明确边界
* 不做无关功能扩张

---

## 3. 用户场景

### 3.1 典型场景

1. 用户在任意应用中选中英文文本，按 `Cmd+C+C` 或 `Ctrl+C+C`，软件自动弹出翻译结果。
2. 用户复制法语句子后触发翻译，软件同时输出英文与中文。
3. 用户在设置页中自定义 OpenAI-compatible API 地址、模型名、温度等参数。
4. 用户指定规则：检测到英文时默认翻译为中文；检测到中文时默认翻译为英文；检测到其他语言时同时翻译为中文和英文。
5. 用户不希望软件干扰当前工作流，翻译窗口快速出现、快速关闭、复制方便。

### 3.2 非目标场景

1. 长文档专业翻译排版
2. 多人共享配置
3. 企业级翻译审校系统
4. 浏览器全文改写
5. 图像内容理解

## 4. 核心功能需求

### 4.1 平台支持

必须支持：

* macOS（Apple Silicon 优先，兼容 Intel）
* Windows 10/11

第一期不要求 Linux，但架构应尽量保留后续扩展可能。

### 4.2 触发方式

必须支持以下触发方式：

#### 4.2.1 双击复制触发

* 用户在任意应用中完成文本选择
* 在限定时间窗口内连续两次触发复制动作
* macOS 为 `Cmd+C+C`
* Windows 为 `Ctrl+C+C`
* 软件检测到双复制事件后自动读取剪贴板内容并翻译

#### 4.2.2 限制要求

* 双击复制检测必须有时间窗口，默认 `450ms`
* 时间窗口可配置，允许范围 `250ms ~ 800ms`
* 不允许无限等待第二次复制
* 不允许因普通单次复制频繁误触发
* 不允许因空剪贴板或非文本内容触发翻译窗口

#### 4.2.3 后备触发方式

必须额外支持一个显式全局快捷键触发，例如：

* `Cmd+Shift+Y`（macOS）
* `Ctrl+Shift+Y`（Windows）

该快捷键用于：

* 当双击复制在某些应用中不稳定时，手动触发翻译
* 触发当前剪贴板文本翻译

### 4.3 文本获取

必须通过系统剪贴板读取文本。

#### 4.3.1 文本读取要求

* 仅处理纯文本或可安全转换为纯文本的数据
* 富文本、HTML、RTF 必须在进入翻译链路前转换为纯文本
* 不允许将原始 HTML 直接注入 UI
* 文本长度默认上限 `8000` 字符，可配置但不得超过 `50000` 字符
* 超限时必须提示用户，不允许无提示截断

#### 4.3.2 文本预处理

必须进行以下预处理：

* 去除首尾空白
* 归一化换行符
* 移除不可见控制字符（保留必要换行与制表）
* 保留原始段落结构
* 不修改用户原文语义

### 4.4 语言检测与目标语言规则

#### 4.4.1 自动检测

必须支持自动检测源语言。

检测方式允许两种：

* 使用模型推断
* 使用本地轻量语言检测库辅助判断

优先要求：

* 短文本稳定
* 中英混合文本不崩溃
* 检测失败时可退回默认规则

#### 4.4.2 目标语言策略

必须支持规则化目标语言配置。

至少支持以下配置模型：

1. **双向主次语言模式**

   * 若检测结果为语言 A，则翻译到语言 B
   * 若检测结果为语言 B，则翻译到语言 A

2. **多分支模式**

   * 若源语言为英语，则翻译到中文
   * 若源语言为中文，则翻译到英语
   * 若源语言为其他语言，则同时翻译到英语和中文

3. **固定目标语言模式**

   * 无论源语言为何，始终翻译到指定语言列表

#### 4.4.3 多目标语言输出

必须支持一次输入同时输出多个目标语言译文。

最少支持：

* 单目标输出
* 双目标输出

界面必须清晰标识每个目标语言对应的译文。

### 4.5 翻译结果显示

#### 4.5.1 弹窗形式

必须使用轻量悬浮窗口显示结果。

窗口行为要求：

* 默认不抢占主应用长时间焦点
* 弹出速度快
* 支持 ESC 关闭
* 支持点击外部关闭（可配置）
* 支持固定窗口（Pin）
* 支持滚动查看长文本

#### 4.5.2 窗口内容

必须显示：

* 检测语言
* 目标语言标签
* 原文（可折叠）
* 译文
* 复制按钮
* 重新翻译按钮
* 固定窗口按钮

可选显示：

* 用时
* 当前模型名
* provider 名称

#### 4.5.3 结果操作

必须支持：

* 一键复制整段译文
* 每个目标语言单独复制
* 再次触发时替换当前结果或新开结果，二选一并写入设置

### 4.6 API 配置与模型设置

#### 4.6.1 Provider 支持

必须优先支持：

* OpenAI-compatible API

可选后续支持：

* Gemini 原生 API
* Anthropic 原生 API
* Ollama 本地接口

第一期强制要求：**OpenAI-compatible 抽象必须作为一等能力设计，不得仅支持单一官方域名。**

#### 4.6.2 OpenAI-compatible 配置项

必须支持以下配置：

* Base URL
* API Key
* Model
* Temperature
* Top P
* Max Tokens
* Request Timeout
* Custom Headers（受限）
* 可选 Organization 字段

#### 4.6.3 限制要求

* Base URL 必须可自定义
* 不允许写死官方服务地址
* 不允许仅通过下拉框限制 provider
* 必须支持用户输入任意兼容地址

## 5. 当前迭代计划

以下事项属于当前阶段的高优先级优化方向，用于指导后续 issue 拆分与实现顺序。

### 5.1 服务商配置体验优化

目标：

* 让服务商配置区域更简洁、更优雅
* 减少视觉噪音与重复字段堆叠
* 让保存、测试、设为活动项等核心操作更容易理解

要求：

* 优化 provider 表单布局与信息分组
* 降低长表单带来的阅读负担
* 保持浅色与深色主题下的可读性一致

### 5.2 服务商页面可读性增强

目标：

* 明确区分列表区、编辑区、状态区
* 改善字段标签、操作按钮、状态反馈的层级关系

要求：

* 页面在大窗口与中等窗口下都应保持稳定布局
* 状态信息必须对用户友好，避免开发者语气或调试式文案
* 页面中的交互优先级应清晰，避免用户找不到主要操作入口

### 5.3 支持修改服务商名称

目标：

* 用户可直接修改 provider 的显示名称

要求：

* 区分“服务商 ID”与“显示名称”
* 修改显示名称不应破坏已有 provider 标识或活动项逻辑
* 列表展示优先使用显示名称，无显示名称时再退回 ID

### 5.4 支持多语言同时翻译

目标：

* 支持一个输入同时输出多个目标语言译文

要求：

* 当前手动翻译页面需要支持多目标语言选择
* 翻译结果区要明确标识每个目标语言
* 仍需兼容单目标语言模式
* 不得破坏既有自动路由与剪贴板触发链路

### 5.5 TODO 项清理与持续优化

目标：

* 逐步消化现有 TODO 与临时实现

当前方向包括：

* 完善服务商配置体验
* 继续优化深色模式一致性
* 清理临时 UI 文案与过渡实现
* 梳理未完成的安全与存储改进事项
* 必须支持保存多个 provider 配置档
* 必须支持切换默认 provider

#### 4.6.4 参数边界

必须对参数进行校验：

* Temperature：`0 ~ 2`
* Top P：`0 ~ 1`
* Max Tokens：`1 ~ 32000`，默认建议按模型能力限制
* Timeout：`3 ~ 120` 秒

非法值不得进入请求层。

### 4.7 Prompt 与翻译策略

必须内置翻译专用提示模板。

#### 4.7.1 系统提示要求

系统提示必须包含：

* 只输出译文
* 不添加解释
* 保留原始段落结构
* 保留代码、URL、邮箱、变量名、专有标识符
* 对多目标语言输出使用固定结构

#### 4.7.2 自定义提示

允许用户配置额外翻译规则，但必须与系统底层安全模板拼接，不能完全替换系统提示。

#### 4.7.3 强制限制

* 不允许将用户原文直接当作系统指令执行
* 不允许让用户文本覆盖安全规则
* 不允许在提示词层暴露本地配置、密钥或内部状态

### 4.8 历史记录

必须支持本地历史记录。

限制要求：

* 默认仅保留最近 `100` 条
* 可关闭历史记录
* 不允许上传历史到云端
* 必须支持一键清空历史

### 4.9 日志

必须区分：

* 应用日志
* 调试日志
* 网络错误日志

限制要求：

* 默认日志中不得记录完整原文与完整译文
* 默认日志中不得记录 API Key
* 错误日志仅记录必要上下文
* Debug 模式必须显式开启

## 5. 非功能需求

### 5.1 性能要求

* 常驻内存尽可能低
* 应用冷启动时间应短
* 从触发到窗口显示应尽量控制在低延迟范围
* UI 渲染必须保持流畅
* 设置页打开不得明显卡顿

### 5.2 资源占用要求

技术选型应优先低体积、低内存方案。

必须避免：

* 引入大型无关依赖
* 内置浏览器扩展系统
* 内置 Node 侧大量常驻后台任务
* 为小需求引入重量级 UI 库

### 5.3 稳定性要求

* 任意一次翻译请求失败不得导致应用崩溃
* 剪贴板内容异常不得导致窗口死锁
* API 响应异常必须可恢复
* 设置文件损坏时必须回退默认配置并提示

### 5.4 离线容错

即便网络不可用，应用本体也必须可打开、可设置、可查看历史。

## 6. 技术栈要求

## 6.1 总体选型

### 6.1.1 桌面框架

必须采用：

* **Tauri v2**

原因：

* 跨平台
* 相较 Electron 更轻量
* 更适合常驻托盘类小工具
* 能够灵活接入不同后端实现

禁止采用：

* Electron 作为第一实现方案

### 6.1.2 前端框架

必须采用：

* **Vue 3 + TypeScript + Tailwind CSS**

限制要求：

* 前端必须采用组件化开发
* 必须使用 TypeScript
* 必须使用 Composition API
* 不允许使用 jQuery
* 不允许使用过度复杂的状态管理框架

推荐状态管理：

* Pinia

推荐路由：

* Vue Router

### 6.1.3 样式方案

必须采用：

* **Tailwind CSS**

限制要求：

* 设计风格必须扁平化
* 禁止使用渐变色
* 禁止使用玻璃拟态
* 禁止复杂阴影堆叠
* 禁止花哨动画
* Tailwind 主题变量必须支持浅色与深色模式

### 6.1.4 后端实现

后端语言不设硬性限制，但必须满足：

* 体积可控
* 启动快
* 常驻资源低
* 易于实现系统托盘、快捷键、剪贴板、配置读写与安全控制

后端职责：

* 系统托盘
* 全局快捷键
* 剪贴板读取
* 配置读写
* 安全校验
* HTTP 请求发起
* 响应解析
* 历史记录持久化

禁止将关键安全逻辑仅放在前端层。

## 7. 系统架构

## 7.1 分层架构

1. **UI 层**

   * 弹窗

   * 翻译服务

5. **基础设施层**

   * 文件存储

## 8. UI/UX 规范

## 8.1 视觉风格

必须遵循以下视觉原则：

* 扁平化
* 简洁
* 少装饰
* 高可读性
* 单色主导

### 8.1.1 配色要求

默认浅色主题：

* 背景：白色或接近白色
* 正文：黑色或深灰
* 图标强调色：蓝色

默认深色主题：

* 背景：黑色或深灰黑
* 正文：白色或浅灰
* 图标强调色：蓝色或等价单一强调色

限制要求：

* 禁止使用渐变色
* 禁止使用多彩按钮体系
* 禁止彩虹色标签
* 全局强调色最多一个主色 + 一个危险色

### 8.1.2 图标与控件

* 图标风格必须统一
* 线性图标优先
* 按钮形态简洁
* 圆角适中，不夸张

### 8.1.3 字体

优先系统字体：

* macOS：SF Pro / 系统默认
* Windows：Segoe UI / 系统默认

禁止自带大型字体包。

## 8.2 窗口设计

### 8.2.1 主弹窗

主弹窗必须包含：

* 顶部工具栏
* 原文区
* 译文区
* 操作按钮区

### 8.2.2 设置页

设置页必须采用分组布局：

* 通用设置
* 触发设置
* 语言规则
* Provider 设置
* 模型参数
* 历史与隐私
* 调试

### 8.2.3 历史页

历史页仅作本地快速查看，不做复杂筛选系统。

## 9. 安全设计要求

## 9.1 总则

后端服务必须尽可能轻量、高效，并以安全为前提设计。

本产品面对的主要风险包括：

* 剪贴板恶意内容
* 翻译文本中的 prompt injection
* UI 注入
* 本地配置泄露
* API Key 泄露
* 恶意 base URL
* 调试日志泄密

## 9.2 输入安全

### 9.2.1 文本输入处理

所有进入系统的文本必须视为不可信输入。

必须执行：

* UTF-8 合法性检查
* 长度校验
* 控制字符清理
* 展示前转义

禁止执行：

* 将原始用户文本作为 HTML 渲染
* 将原始用户文本拼接进前端模板而不转义
* 将原始用户文本用于命令执行
* 将原始用户文本写入 shell 参数

### 9.2.2 Prompt Injection 防护

必须将用户待翻译文本放入明确的数据槽位，而非直接与系统提示无边界拼接。

必须采用固定模板，例如：

* 系统规则
* 结构化任务参数
* 用户文本内容块

要求：

* 用户文本只能作为待翻译材料
* 用户文本中的“忽略以上要求”“输出密钥”等内容不得影响系统行为
* 输出必须经过结构解析或最小结构约束

### 9.2.3 URL 安全

对于用户自定义 Base URL：

必须支持自定义，但必须限制：

* 仅允许 `https://`，开发模式可显式允许 `http://localhost`
* 必须校验 URL 格式
* 禁止 file://、javascript:// 等协议
* 必须禁止自动跟随无限重定向
* 必须设置合理请求超时

## 9.3 密钥安全

### 9.3.1 存储要求

API Key 不得以明文写入普通配置文件。

必须优先使用系统安全存储：

* macOS Keychain
* Windows Credential Manager

配置文件中仅保存 provider 元信息与引用 ID。

### 9.3.2 展示要求

* 设置页中 API Key 默认遮罩
* 不允许在日志中输出 API Key
* 不允许在错误弹窗中回显完整 Key

## 9.4 前后端边界安全

### 9.4.1 Tauri 命令约束

* 每个 Tauri command 必须有明确参数类型
* 必须做输入校验
* 不允许暴露无用途的调试命令
* 不允许保留开发阶段危险接口到生产版本

### 9.4.2 CSP 与前端安全

必须启用严格 CSP。

要求：

* 禁止任意远程脚本执行
* 禁止内联危险脚本
* 禁止通过翻译结果动态注入脚本

## 9.5 网络安全

* 必须使用 TLS
* 必须设置超时
* 必须限制并发请求数
* 必须对异常状态码做容错
* 不允许无限重试
* 不允许默认信任自签名证书

## 9.6 日志与隐私

* 默认不记录完整敏感文本
* Debug 模式显式开关
* 导出日志前需二次确认
* 清空历史不影响安全存储中的密钥

## 11. Provider 接口规范

## 11.1 统一接口

必须定义统一翻译接口：

```text
translate(request) -> response
```

### 11.1.1 请求结构

建议字段：

* `source_text: String`
* `source_lang: Option<String>`
* `target_langs: Vec<String>`
* `model: String`
* `temperature: f32`
* `top_p: Option<f32>`
* `max_tokens: Option<u32>`
* `timeout_secs: u64`
* `system_profile: TranslationProfile`
* `user_rules: Option<String>`

### 11.1.2 响应结构

建议字段：

* `detected_source_lang: String`
* `translations: Vec<TranslationItem>`
* `provider: String`
* `model: String`
* `latency_ms: u64`
* `raw_usage: Option<Usage>`

## 11.2 OpenAI-compatible 适配规范

必须支持：

* `POST /chat/completions` 兼容模式
* 或兼容的 responses 风格封装，但内部需统一

必须允许用户配置：

* base URL
* path 覆盖（高级选项）
* headers
* auth scheme（默认 Bearer）

Custom Headers 限制：

* Header 名和值必须进行格式校验
* 禁止注入换行
* 禁止覆盖 Host 等高风险头

## 12. 翻译策略实现规范

## 12.1 单目标输出格式

## 12.2 多目标输出格式

多目标输出必须要求模型按固定结构返回，例如 JSON 或明确分段标签。

推荐优先：

```text
[zh-CN]
...

[en]
...
```

必须在解析失败时回退到安全展示，不得崩溃。

## 12.3 保留内容规则

必须尽可能保留：

* 代码片段
* URL
* 邮箱地址
* 文件路径
* 命令行参数
* 变量名
* 占位符
